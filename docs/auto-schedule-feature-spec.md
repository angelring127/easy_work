# 자동 스케줄 기능 명세서

이 문서는 Workeasy의 자동 스케줄 등록 기능을 구현하기 위한 제품/개발 명세서다. 목표는 매장 영업 시간과 근무 항목을 기준으로 직원에게 스케줄을 자동 배정하되, 법적/운영상 반드시 지켜야 하는 조건은 절대 위반하지 않고, 선호 조건은 매장별 우선순위와 점수 방식으로 반영하는 것이다.

## 1. 기능 목표

자동 스케줄 기능은 매장 관리자가 특정 기간의 스케줄을 빠르게 생성할 수 있도록 돕는다.

주요 목표:

- 매장 영업시간 동안 빈 근무 시간이 없도록 스케줄을 채운다.
- 근무 항목에 필요한 직무를 가진 직원만 배정한다.
- 주간/월간 최대 근무 시간, 출근 불가, 다른 브랜치 출근 충돌 같은 필수 조건은 절대 위반하지 않는다.
- 직원 희망 주 근무 시간, 휴일 희망일, 출근 선호일은 매장 설정의 우선순위에 따라 점수로 반영한다.
- 직원별 자동 배정 우선순위를 매장 설정에서 직접 정할 수 있게 한다.
- 자동 배정 결과가 완벽하지 않을 경우, 어떤 조건 때문에 미배정 또는 부족 커버가 발생했는지 관리자에게 설명한다.

## 2. 사용자 범위

자동 스케줄 실행 권한:

- `MASTER`
- `SUB_MANAGER`

자동 스케줄 설정 권한:

- `MASTER`
- 필요하면 이후 `SUB_MANAGER`에게도 위임 가능

자동 스케줄 대상자:

- 해당 매장에 활성 상태로 속한 직원
- 자동 배정에 필요한 직무 역할을 가진 직원
- 비활성 또는 삭제된 직원은 자동 배정 후보에서 제외

## 3. 매장 설정에 추가할 탭

매장 설정 화면에 자동 스케줄 관련 탭을 추가한다.

권장 탭 구조:

- `자동 스케줄 조건`
- `직원 배정 우선순위`

현재 매장 설정 화면은 `/[locale]/stores/[id]/edit`에서 매장 정책, 영업시간, 근무 항목, 직무 역할 등을 다룬다. 자동 스케줄 설정도 같은 화면 안의 탭으로 추가하는 것이 자연스럽다.

## 4. 자동 스케줄 조건 분류

자동 배정 조건은 `필수 조건`과 `점수 조건`으로 나눈다.

### 4.1 필수 조건

필수 조건은 매장에서 순서를 바꾸거나 끌 수 없다. 이 조건을 위반하는 후보는 자동 배정 후보에서 즉시 제외한다.

| 조건 | 설명 | 처리 방식 |
| --- | --- | --- |
| 주간 최대 근무 시간 | 배정 후 직원의 주간 근무 시간이 매장 설정의 최대 주간 근무 시간을 넘으면 안 된다. | 후보 제외 |
| 월간 최대 근무 시간 | 배정 후 직원의 월간 근무 시간이 매장 설정의 최대 월간 근무 시간을 넘으면 안 된다. | 후보 제외 |
| 출근 불가 | 직원이 해당 날짜 또는 해당 시간대에 출근 불가로 등록한 경우 배정하지 않는다. | 후보 제외 |
| 다른 브랜치 출근 충돌 | 같은 직원이 다른 매장/브랜치에서 같은 시간에 이미 근무하는 경우 배정하지 않는다. | 후보 제외 |
| 직무 미보유 | 근무 항목에 필요한 직무를 직원이 가지고 있지 않으면 배정하지 않는다. | 후보 제외 |
| 오프닝 앵커 근무 | 영업 시작 시간에는 오프닝 가능 직무를 가진 직원 1명이 먼저 출근해야 한다. | 선행 배정 |
| 오전/오후 최대 인원 | 근무 항목 시간대가 오전/오후 인원 제한을 초과하면 배정하지 않는다. | 후보 제외 |
| 영업시간 빈틈 없음 | 영업시간 전체가 근무 항목과 배정 인원으로 커버되어야 한다. | 결과 검증 |

### 4.2 점수 조건

점수 조건은 필수 조건을 통과한 후보들 사이에서 더 적합한 직원을 고르기 위한 조건이다. 매장 설정에서 드래그로 우선순위를 조정할 수 있다.

| 조건 | 설명 | 기본 점수 방향 |
| --- | --- | --- |
| 희망 주 근무 시간 | 직원의 희망 주 근무 시간에 가깝게 배정한다. | 초과하지 않고 목표에 가까우면 가점 |
| 휴일 희망일 | 직원이 휴일로 희망한 날짜에는 가능한 배정하지 않는다. | 휴일 희망일이 아니면 가점 |
| 출근 선호일 | 직원이 선호하는 요일에는 우선 배정한다. | 선호 요일이면 가점 |

기본 우선순위 추천:

1. 희망 주 근무 시간
2. 휴일 희망일
3. 출근 선호일

## 5. 조건 우선순위 점수 방식

자동 배정은 필수 조건을 먼저 적용한 후, 남은 후보에게 점수를 계산한다.

기본 가중치:

| 순위 | 점수 |
| --- | --- |
| 1순위 | 30점 |
| 2순위 | 20점 |
| 3순위 | 10점 |

예시:

- 매장 설정 우선순위가 `출근 선호일 > 희망 주 근무 시간 > 휴일 희망일`이면, 출근 선호일 조건을 만족하는 후보에게 30점, 희망 주 근무 시간 조건을 잘 만족하는 후보에게 최대 20점, 휴일 희망일 조건을 만족하는 후보에게 10점을 부여한다.

점수 계산 원칙:

- 조건을 완전히 만족하면 해당 순위의 전체 점수를 부여한다.
- 조건을 일부만 만족하는 경우 부분 점수를 줄 수 있다.
- 필수 조건을 통과하지 못한 후보는 점수 계산 대상이 아니다.

희망 주 근무 시간 점수 예시:

- 배정 후 주간 시간이 희망 주 근무 시간을 넘으면 감점 또는 0점
- 배정 후 주간 시간이 희망 주 근무 시간에 가까우면 높은 점수
- 희망 주 근무 시간이 없는 직원은 중립 점수 또는 0점

휴일 희망일 점수 예시:

- 해당 날짜가 휴일 희망일이 아니면 전체 점수
- 해당 날짜가 휴일 희망일이면 0점

출근 선호일 점수 예시:

- 해당 요일이 출근 선호일이면 전체 점수
- 선호일 정보가 없거나 선호일이 아니면 0점

## 6. 오프닝 앵커 근무

오프닝 근무는 일반적인 오전/오후 인원 채우기보다 먼저 처리해야 하는 특수 조건이다. 예를 들어 매장이 10시에 열리면, 10시부터는 `오프닝` 근무 항목 또는 오프닝 가능 직무를 가진 직원 중 최소 1명이 먼저 출근해서 오픈 준비를 담당한다. 그 이후 11시부터 풀타임 또는 하프타임 직원이 추가로 들어와 일반 영업 커버를 이어간다.

### 6.1 기본 모델

오프닝은 별도의 `앵커 슬롯`으로 취급한다. 단, 시간과 필요 인원은 코드에 고정하지 않고 매장 설정에서 수정할 수 있어야 한다.

예시:

| 시간 | 슬롯 유형 | 필요 인원 | 후보 조건 |
| --- | --- | --- | --- |
| 10:00-11:00 | 오프닝 앵커 | 1명 | 오프닝 근무 항목 또는 오프닝 가능 직무 보유 |
| 11:00-18:00 | 일반 영업 커버 | 근무 항목별 필요 인원 | 풀타임/하프타임/기타 해당 직무 보유 |

이 구조에서는 자동 배정 엔진이 하루를 처리할 때 오프닝 앵커 슬롯을 먼저 배정하고, 그 다음 일반 근무 슬롯을 배정한다.

### 6.2 시스템 설정 가능 항목

오프닝 정책은 매장마다 다를 수 있으므로 시스템 설정값으로 관리한다.

설정 가능 항목:

| 설정 | 설명 | 기본값 |
| --- | --- | --- |
| 오프닝 앵커 사용 여부 | 자동 배정에서 오프닝 슬롯을 별도 필수 조건으로 생성할지 결정한다. | 사용 |
| 오프닝 시작 기준 | 영업 시작 시간과 같게 할지, 별도 시간을 지정할지 결정한다. | 영업 시작 시간 |
| 오프닝 시작 시간 | 별도 시작 시간을 사용할 때 적용한다. | 매장 영업 시작 시간 |
| 오프닝 종료 시간 | 오프닝 단독 구간이 끝나는 시간이다. 예: 11:00 | 영업 시작 + 60분 |
| 오프닝 필요 인원 | 오프닝 구간에 필요한 최소 인원이다. | 1명 |
| 오프닝 가능 근무 항목 | 오프닝 앵커로 인정할 근무 항목을 선택한다. | `오프닝` 근무 항목 |
| 오프닝 가능 직무 | 오프닝 앵커 후보가 되기 위해 필요한 직무를 선택한다. | 매장 설정값 없음 |
| 오프닝 미배정 시 처리 | 오프닝 배정 실패 시 전체 자동 배정을 중단할지, 일반 배정은 계속할지 결정한다. | 일반 배정 계속 + 필수 경고 |

예를 들어 어떤 매장은 10:00-11:00에 1명만 오프닝하면 되고, 어떤 매장은 09:30-11:00에 2명이 필요할 수 있다. 자동 스케줄 엔진은 이 차이를 코드 수정 없이 매장 설정값으로 읽어 처리해야 한다.

### 6.3 오프닝 후보 조건

오프닝 후보는 다음 조건을 모두 만족해야 한다.

- 해당 매장의 활성 직원이다.
- 오프닝 근무 항목에 필요한 직무를 가지고 있다.
- 영업 시작 시간에 출근할 수 있다.
- 해당 날짜 또는 오프닝 시간대에 출근 불가가 아니다.
- 다른 브랜치에서 같은 시간대에 근무 중이 아니다.
- 배정 후 주간/월간 최대 근무 시간을 초과하지 않는다.
- 오전 최대 근무 인원 제한을 초과하지 않는다.

오프닝이 가능한 직원이 없으면 해당 날짜는 `오프닝 미배정` 상태로 리포트하고, 일반 슬롯 배정은 정책에 따라 계속 진행할 수 있다. 단, 최종 결과에는 영업 시작 커버 실패가 필수 경고로 표시되어야 한다.

### 6.4 오프닝과 일반 근무의 연결

오프닝 직원은 두 가지 방식 중 하나로 배정할 수 있다.

권장 방식은 `오프닝 근무 항목을 별도 배정`하는 것이다.

- 10:00-11:00 오프닝 근무 항목을 먼저 배정한다.
- 같은 직원이 11시 이후에도 계속 근무할 수 있으면, 일반 근무 항목 후보에도 다시 포함할 수 있다.
- 다만 같은 직원에게 시간이 겹치는 근무 항목을 중복 배정하지 않는다.
- 10:00-18:00처럼 오프닝부터 일반 근무까지 이어지는 근무 항목이 있다면, 그 항목은 오프닝 앵커와 일반 커버를 동시에 만족할 수 있다.

대안 방식은 `오프닝 플래그를 가진 일반 근무 항목`이다.

- 풀타임 근무 항목에 `오프닝 가능` 속성을 둔다.
- 10시에 시작하는 풀타임 근무자가 있으면 오프닝 앵커를 만족한 것으로 본다.
- 11시에 시작하는 풀타임/하프타임은 오프닝 앵커를 만족하지 못한다.

초기 구현은 기존 근무 항목 구조와 잘 맞는 쪽을 선택해야 한다. 명세 기준으로는 두 방식을 모두 허용하되, 자동 배정 엔진은 `영업 시작 시간에 오프닝 가능 직원 1명 이상`이라는 결과 조건을 반드시 검증한다.

### 6.5 점수 적용

오프닝 앵커도 필수 조건을 통과한 후보 중에서 점수 방식으로 선택한다.

선택 순서:

1. 오프닝 가능 후보만 남긴다.
2. 출근 불가, 다른 브랜치 충돌, 주/월 최대 근무시간을 적용한다.
3. 매장 설정의 조건 우선순위 점수를 계산한다.
4. 직원 배정 우선순위 보조 점수를 더한다.
5. 동점이면 주간 근무시간, 월간 근무시간, 최근 배정일 순으로 결정한다.

오프닝은 영업 시작을 책임지는 슬롯이므로, 일반 슬롯보다 먼저 배정되어야 한다. 그래야 오프닝에 적합한 직원이 뒤의 일반 슬롯에 먼저 소모되는 문제를 줄일 수 있다.

## 7. 직원 배정 우선순위

매장 설정에 직원별 자동 배정 우선순위 탭을 추가한다.

### 7.1 UI 요구사항

직원 목록을 드래그해서 1순위부터 끝까지 직접 정렬한다.

표시 정보:

- 순위 번호
- 직원 이름
- 역할 또는 직무
- 상태
- 이번 주 배정 시간
- 이번 달 배정 시간

상태 처리:

- 새 직원은 자동으로 목록 맨 아래에 추가한다.
- 비활성 직원은 자동 배정 후보에서 제외한다.
- 삭제된 직원은 자동 배정 후보에서 제외한다.
- 비활성/삭제 직원이 기존 우선순위 목록에 남아 있어도 자동 배정에는 사용하지 않는다.

### 7.2 점수 반영 방식

직원 우선순위는 조건 점수 이후의 보조 점수로 반영한다.

권장 방식:

- 직원 목록이 위에 있을수록 높은 보조 점수를 준다.
- 전체 직원 수에 따라 점수를 정규화한다.
- 조건 점수보다 강하게 작동하지 않도록 보조 점수의 최대값을 제한한다.

예시:

- 조건 점수 최대치: 60점
- 직원 우선순위 보조 점수 최대치: 10점
- 1순위 직원: 10점
- 마지막 순위 직원: 0점 또는 1점

이 방식은 관리자가 선호 직원을 위에 둘 수 있게 하면서도, 필수 조건과 주요 선호 조건보다 우선순위가 과도하게 커지는 것을 막는다.

## 8. 동점 처리 기준

조건 점수와 직원 우선순위 보조 점수까지 같으면 다음 기준으로 결정한다.

1. 이번 주 배정 시간이 적은 직원
2. 이번 달 배정 시간이 적은 직원
3. 최근 배정된 날짜가 오래된 직원
4. 그래도 같으면 안정적인 정렬 기준 사용

안정적인 정렬 기준은 `priority_rank`, `user_name`, `user_id`처럼 매번 같은 결과가 나오도록 정한다. DB 조회 순서에 따라 결과가 흔들리면 안 된다.

## 9. 자동 배정 처리 흐름

자동 스케줄 실행 흐름:

1. 매장, 시작일, 종료일을 입력받는다.
2. 매장 영업시간을 조회한다.
3. 근무 항목과 근무 항목별 필요 직무를 조회한다.
4. 활성 직원과 직원별 직무를 조회한다.
5. 직원별 출근 불가, 휴일 희망일, 출근 선호일, 희망 주 근무 시간을 조회한다.
6. 기존 스케줄 배정 내역을 조회한다.
7. 다른 브랜치의 같은 시간대 배정 내역을 조회한다.
8. 영업 시작 시간의 오프닝 앵커 슬롯을 먼저 만든다.
9. 영업시간을 커버해야 하는 일반 근무 슬롯으로 나눈다.
10. 오프닝 앵커 슬롯을 먼저 배정한다.
11. 각 일반 슬롯마다 필수 조건을 통과한 후보를 만든다.
12. 후보별 조건 점수와 직원 우선순위 보조 점수를 계산한다.
13. 최고 점수 후보를 배정한다.
14. 주간/월간 누적 시간을 즉시 갱신한다.
15. 모든 슬롯 처리 후 오프닝 커버와 영업시간 커버 여부를 검증한다.
16. 결과를 저장하거나, 미리보기 모드일 경우 저장하지 않고 결과만 반환한다.
17. 생성된 배정, 미배정 슬롯, 경고, 조건별 실패 사유를 반환한다.

## 10. 미리보기와 실제 등록

자동 스케줄은 `미리보기`와 `등록`을 분리하는 것이 좋다.

미리보기:

- 실제 DB에 저장하지 않는다.
- 생성 예정 배정 목록을 보여준다.
- 필수 조건으로 제외된 후보 사유를 요약한다.
- 영업시간 중 커버되지 않은 구간을 보여준다.
- 조건 점수와 직원 우선순위 반영 결과를 확인할 수 있게 한다.

등록:

- 미리보기 결과를 확정 저장한다.
- 등록 직전 기존 스케줄이 바뀌었을 가능성을 다시 검증한다.
- 충돌이 생겼으면 저장을 중단하고 재실행을 요청한다.

## 11. 실패 처리와 결과 메시지

자동 배정은 완벽한 결과를 만들지 못할 수 있다. 이때 아무 설명 없이 실패하면 관리자가 수정할 수 없으므로, 실패 사유를 구조화해서 보여준다.

결과 유형:

- `created`: 생성된 배정 수
- `skipped`: 건너뛴 슬롯 수
- `uncovered`: 영업시간 중 커버되지 않은 구간
- `warnings`: 등록은 가능하지만 확인이 필요한 항목
- `blockingReasons`: 필수 조건 때문에 배정할 수 없었던 사유

대표 실패 사유:

- 해당 시간대에 필요한 직무를 가진 직원이 없다.
- 모든 후보가 주간 최대 근무 시간을 초과한다.
- 모든 후보가 월간 최대 근무 시간을 초과한다.
- 모든 후보가 출근 불가 상태다.
- 모든 후보가 다른 브랜치에서 이미 근무 중이다.
- 오전/오후 최대 인원을 초과한다.
- 오프닝 가능 직무를 가진 직원이 없다.
- 오프닝 가능 직원이 모두 출근 불가 또는 시간 충돌 상태다.
- 근무 항목이 영업시간 전체를 커버하지 못한다.

## 12. 데이터 모델 제안

현재 코드에는 매장 정책, 직원 선호 요일, 출근 불가, 근무 항목, 직무 역할, 스케줄 배정 테이블이 이미 존재한다. 자동 스케줄 설정을 안정적으로 저장하기 위해 별도 설정 테이블을 추가하는 것을 권장한다.

### 12.1 자동 스케줄 조건 우선순위

테이블 예시: `store_auto_schedule_condition_priorities`

| 컬럼 | 설명 |
| --- | --- |
| `id` | 기본 키 |
| `store_id` | 매장 ID |
| `condition_key` | `desired_weekly_hours`, `day_off_preference`, `preferred_weekday` |
| `priority_rank` | 1부터 시작하는 우선순위 |
| `weight` | 점수 가중치 |
| `created_at` | 생성 시각 |
| `updated_at` | 수정 시각 |

제약:

- `store_id + condition_key`는 유니크
- `store_id + priority_rank`는 유니크

### 12.2 직원 자동 배정 우선순위

테이블 예시: `store_auto_schedule_user_priorities`

| 컬럼 | 설명 |
| --- | --- |
| `id` | 기본 키 |
| `store_id` | 매장 ID |
| `user_id` | 직원 ID |
| `priority_rank` | 1부터 시작하는 직원 순위 |
| `created_at` | 생성 시각 |
| `updated_at` | 수정 시각 |

제약:

- `store_id + user_id`는 유니크
- `store_id + priority_rank`는 유니크

운영 규칙:

- 신규 직원이 추가되면 해당 매장의 우선순위 목록 맨 아래에 자동 추가한다.
- 기존 직원이 비활성화되면 우선순위 데이터는 유지하되 자동 배정 후보에서는 제외한다.
- 직원이 삭제되면 우선순위 데이터는 정리하거나, 히스토리 보존 정책에 따라 비활성 참조로 남긴다.

### 12.3 오프닝 설정

오프닝 조건은 시스템적으로 수정 가능해야 한다. 초기 구현에서는 매장별 오프닝 정책 테이블을 두고, 근무 항목에는 오프닝 앵커로 사용할 수 있는지 표시하는 속성을 추가하는 방식을 권장한다.

테이블 예시: `store_auto_schedule_opening_policies`

| 컬럼 | 설명 |
| --- | --- |
| `id` | 기본 키 |
| `store_id` | 매장 ID |
| `enabled` | 오프닝 앵커 조건 사용 여부 |
| `start_source` | `business_open` 또는 `custom` |
| `custom_start_min` | 별도 시작 시간을 사용할 때의 시작 분 |
| `end_min` | 오프닝 단독 구간 종료 분 |
| `required_headcount` | 오프닝 필요 인원 |
| `failure_mode` | `warn_and_continue` 또는 `block_commit` |
| `created_at` | 생성 시각 |
| `updated_at` | 수정 시각 |

근무 항목 권장 속성:

| 컬럼 | 설명 |
| --- | --- |
| `is_opening_item` | 이 근무 항목이 오프닝 앵커로 사용될 수 있는지 |
| `opening_credit` | 이 근무 항목이 오프닝 커버를 충족하는지 |

오프닝 가능 직무는 기존 `work_item_required_roles` 구조를 재사용하거나, 오프닝 정책에 별도 required role 목록을 둘 수 있다.

초기 구현에서는 매장 정책에는 `enabled`, `start_source`, `custom_start_min`, `end_min`, `required_headcount`, `failure_mode`를 저장하고, 근무 항목에는 `is_opening_item`을 추가하는 방식이 단순하다.

## 14. UI 요구사항

### 14.1 자동 스케줄 실행 UI

스케줄 화면에 자동 배정 버튼을 제공한다.

권장 흐름:

1. 관리자가 주차를 선택한다.
2. `자동 배정` 버튼을 누른다.
3. 미리보기 모달이 열린다.
4. 생성 예정 스케줄과 미배정 구간을 확인한다.
5. 문제가 없으면 `등록`을 누른다.

미리보기 모달에 표시할 정보:

- 생성 예정 배정 수
- 오프닝 배정 여부
- 커버되지 않은 시간대
- 필수 조건 때문에 제외된 주요 사유
- 직원별 예상 주간/월간 근무 시간
- 조건 우선순위와 직원 우선순위가 적용된 결과 요약

### 14.2 조건 우선순위 설정 UI

매장 설정의 `자동 스케줄 조건` 탭에서 세 조건을 드래그로 정렬한다.

표시 항목:

- 희망 주 근무 시간
- 휴일 희망일
- 출근 선호일

필수 조건은 같은 탭 하단에 읽기 전용으로 표시한다.

읽기 전용 필수 조건:

- 주간 최대 근무 시간
- 월간 최대 근무 시간
- 출근 불가
- 다른 브랜치 출근 충돌
- 직무 미보유
- 오프닝 앵커 근무
- 오전/오후 최대 인원
- 영업시간 빈틈 없음

### 14.3 직원 우선순위 설정 UI

매장 설정의 `직원 배정 우선순위` 탭에서 직원 목록을 드래그로 정렬한다.

요구사항:

- 드래그 후 저장 버튼으로 반영한다.
- 검색 또는 필터가 있으면 직원 수가 많아도 관리하기 쉽다.
- 비활성 직원은 배정 제외 상태를 명확히 표시한다.
- 새 직원은 자동으로 목록 맨 아래에 들어온다는 안내를 제공한다.

### 14.4 오프닝 설정 UI

근무 항목 설정 또는 자동 스케줄 조건 탭에서 오프닝 관련 설정을 제공한다.

권장 설정:

- 오프닝 앵커 사용 여부
- 오프닝 필요 인원
- 오프닝 시작 시간
- 오프닝 단독 종료 시간
- 오프닝 가능 근무 항목 또는 오프닝 가능 직무
- 오프닝 실패 시 처리 방식

예시 설정:

- 오프닝 앵커 사용: 켬
- 오프닝 필요 인원: 1명
- 오프닝 시작 시간: 10:00
- 오프닝 단독 종료 시간: 11:00
- 오프닝 가능 근무 항목: `오프닝`
- 오프닝 실패 시 처리: 일반 배정 계속 + 필수 경고

UI는 10시/11시를 고정 문구로 보여주지 않고, 매장 영업시간과 설정값을 읽어 현재 정책을 표시해야 한다. 예를 들어 영업 시작 시간이 09:00인 매장은 기본 오프닝 시작 시간도 09:00으로 계산된다.

## 15. 알고리즘 개요

자동 배정 알고리즘은 다음 순서로 구현한다.

```text
for each date in range:
  build opening anchor slots from business opening time
  build required slots from business hours and work items

  for each opening slot:
    candidates = active users with opening-capable role or opening work item
    candidates = remove users violating hard constraints
    selected = score and select best candidate
    assign selected user
    update in-memory weekly/monthly counters

  for each slot:
    candidates = active users with required role
    candidates = remove users violating hard constraints

    if candidates is empty:
      record uncovered slot and blocking reasons
      continue

    scoredCandidates = candidates.map(user => {
      conditionScore = calculateConditionScore(user, slot, storePriority)
      userPriorityScore = calculateUserPriorityScore(user, storeUserPriority)
      fairnessData = current weekly/monthly hours and last assigned date
      return total score and tie-breaker data
    })

    selected = sort by:
      total score desc
      weekly assigned minutes asc
      monthly assigned minutes asc
      last assigned date asc
      stable user key asc

    assign selected user
    update in-memory weekly/monthly counters

validate final coverage
return assignments, warnings, uncovered slots
```

2단계: 자동 배정 엔진 분리

- 현재 `/api/schedule/auto-assign` 안의 단순 배정 로직을 `src/lib/schedule/auto-assign` 같은 도메인 모듈로 분리
- 오프닝 앵커 슬롯 생성 함수 작성
- 필수 조건 필터링 함수 작성
- 점수 계산 함수 작성
- 동점 처리 함수 작성
- 결과 리포트 타입 작성

3단계: 미리보기와 등록

- 미리보기 API 추가
- 등록 API에서 최종 재검증
- 스케줄 화면에 미리보기 모달 추가

4단계: 검증과 테스트

- 단위 테스트로 필수 조건과 점수 계산 검증
- API 테스트로 미리보기/등록 결과 검증
- Playwright로 매장 설정 탭, 드래그 저장, 자동 배정 실행 흐름 검증

## 17. 완료 기준

이 기능은 다음 조건을 만족하면 완료로 본다.

- 매장 설정에서 조건 우선순위를 드래그로 변경하고 저장할 수 있다.
- 매장 설정에서 직원 자동 배정 우선순위를 드래그로 변경하고 저장할 수 있다.
- 매장 설정에서 오프닝 사용 여부, 시간, 필요 인원, 가능 근무 항목, 실패 처리 방식을 수정할 수 있다.
- 새 직원은 우선순위 목록 맨 아래에 자동 추가된다.
- 비활성/삭제 직원은 자동 배정 후보에서 제외된다.
- 오프닝 앵커를 사용하는 매장은 영업 시작 시간에 오프닝 가능 직원이 먼저 배정된다.
- 자동 배정은 필수 조건을 위반하지 않는다.
- 점수 조건은 매장 설정 우선순위에 따라 후보 선택에 반영된다.
- 직원 우선순위는 조건 점수 이후의 보조 점수로 반영된다.
- 동점이면 주간 근무시간, 월간 근무시간, 최근 배정일 순서로 공정하게 결정된다.
- 영업시간 중 커버되지 않은 구간이 있으면 결과에 명확히 표시된다.
- 자동 배정 미리보기와 실제 등록이 분리되어 있다.
- `npm run lint`, `npx tsc --noEmit`, `npm run build`가 통과한다.

## 18. 남은 결정 사항

아래 항목은 구현 전에 추가로 결정하면 좋다.

- 오전/오후 구분 기준을 매장 설정의 `morning_cutoff_min` 같은 정책으로만 볼지, 근무 항목 시간대별로 더 세분화할지
- 영업시간 빈틈 없음의 최소 단위를 몇 분으로 볼지
- 오프닝을 별도 근무 항목으로 둘지, 일반 근무 항목의 `오프닝 가능` 속성으로 둘지
- 오프닝 단독 구간을 항상 10:00-11:00처럼 고정할지, 매장별로 조정 가능하게 할지
- 희망 주 근무 시간 초과를 점수 조건으로만 볼지, 특정 초과 폭 이상은 필수 차단할지
- 다른 브랜치 출근 충돌을 같은 Supabase 프로젝트의 모든 매장 기준으로 볼지, 같은 회사/조직 그룹 안의 매장으로 제한할지
- 미리보기 결과를 임시 저장할지, 매번 실시간 계산할지
